# AI-Driven UI Test Automation
## From Manual Authoring to Intelligent Orchestration

---

# The Problem — How QA Works Today

> Every test run starts with a human opening a spreadsheet.

### The Manual Pipeline

```
TestCases.xlsx
    │
    ▼  (QA reads steps manually)
AdvancedActions.xlsx
    │
    ▼  (QA hand-assembles xlsx script with selenium commands
        and AdvancedActions by copying action names + locators)
script.xlsx ──(compiled to)──▶ script.json
    │
    ▼  (QA runs AutoQA command)
AutoQA (Java Selenium Engine)
    │
    ▼  (AutoQA generates xlsx output report)
QA validates test report generated by AutoQA
```

---

# Manual Workflow — Pain Points

| Pain Point | Impact |
|---|---|
| QA must know every element locator by name | High onboarding time |
| Manually copying action names from AdvancedActions.xlsx | Slow, error-prone |
| No validation until runtime failure | Wasted execution time |
| After Angular upgrade, locators break silently | Test suite rot |
| Results require manual validation of AutoQA xlsx report | Time-consuming interpretation |
| One person's knowledge, not team-shareable | Key-person dependency |

> A single test case with 8 steps takes **30–60 minutes** to script manually.

# Architecture 1 — Python + Gemini Orchestrator

### Designed for: CI pipelines, batch runs, audit-critical workflows

```
                  TestCases.xlsx
                       │
                       ▼
              python main.py --tc TC-001
                       │
           ┌───────────┼───────────┐
           ▼           ▼           ▼
      context_loader  llm_client  tc_loader
      (XLSX → text)  (Gemini API) (steps)
           │           │
           └─────┬─────┘
                 ▼
          JSON Script (attempt 1)
                 │
                 ▼
         script_validator.py  ── ERRORS? ──→ refine (up to 3x)
                 │ VALID
                 ▼
         java_runner.py (IPC → AutoQA)
                 │
         ┌───────┴────────┐
         ▼                ▼
  STEP_RESULT         HELP_REQUEST ──→ LLM heals mid-run
         │
         ▼
   analyze_execution
         │
         ▼
   test-results/ audit trail + xlsx report
```

---

# Architecture 1 — Key Features

| Feature | Detail |
|---|---|
| **LLM** | Google Gemini 2.5 Flash via API |
| **Generation loop** | Up to 3 attempts with validate → refine feedback |
| **Execution retry** | Up to 2 retries with execution trace feedback |
| **Self-healing** | Live IPC: AutoQA sends HELP_REQUEST → LLM → SOLUTION mid-run |
| **Output** | JSON script + xlsx report generated by AutoQA |
| **Audit trail** | Every prompt, response, run trace saved to test-results/ |
| **Entry point** | `cd python-orchestrator && python main.py --tc TC-001` |
| **Deployment** | Single server, REST-exposable, no per-user licence |

# The Shared Handoff Point

> Both AI approaches write the same JSON. AutoQA reads it and produces the same xlsx output.

```json
{
  "testCaseId": "TC-001",
  "requirementId": "REQ-001",
  "steps": [
    { "tcStep": 1, "action": "LoginAsMaker",        "locator": "", "test_input": "", "output": "" },
    { "tcStep": 2, "action": "CreateBuyTrade",       "locator": "", "test_input": "sector=Technology,ticker=AAPL,quantity=100,accountType=Cash,timeInForce=Day Order", "output": "" },
    { "tcStep": 3, "action": "ASSERT_TEXT",          "locator": "$queue-pending-count", "test_input": "Pending: 1", "output": "" },
    { "tcStep": 4, "action": "Logout",               "locator": "", "test_input": "", "output": "" },
    { "tcStep": 5, "action": "LoginAsChecker",       "locator": "", "test_input": "", "output": "" },
    { "tcStep": 6, "action": "ApproveTrade",         "locator": "", "test_input": "", "output": "" },
    { "tcStep": 8, "action": "VerifyDashboardCount", "locator": "", "test_input": "expectedCount=1", "output": "" }
  ]
}
```

> TC-001 (8 test steps) → 7 JSON steps in under 30 seconds. Validated. AutoQA produces the xlsx report.

# Macro-First Composition — The Key Insight

### Before: QA builds xlsx scripts from primitive selenium commands

```
OPEN_URL → TYPE username → TYPE password → CLICK submit →
WAIT_VISIBLE dashboard → CLICK trading menu → CLICK new trade →
WAIT_VISIBLE form → SELECT side → SELECT sector → SELECT ticker →
WAIT_HIDDEN price-spinner → TYPE quantity → CLICK submit →
... (25+ primitive steps per test case, hand-entered in xlsx)
```

### After: AI uses reusable macros from AdvancedActions.xlsx

```
LoginAsMaker → CreateBuyTrade → Logout → LoginAsChecker → ApproveTrade → VerifyDashboardCount
(6 steps, each a macro expanding to its primitives at AutoQA runtime)
```

> Adding a new macro is itself AI-assisted: `/create-advanced-action <description>`

# Self-Healing — Python Orchestrator

> AutoQA calls home mid-execution when a locator fails.

```
AutoQA executing step 12: CLICK on $trade-submit
      │
      ▼  (element not found)
HELP_REQUEST ──→ { "step": 12, "error": "NoSuchElementException", "locator": "$trade-submit" }
      │
      ▼  (Python sends to Gemini)
Gemini: "Try $trade-submit-btn — testid may have been renamed"
      │
      ▼
SOLUTION ──→ { "action": "CLICK", "locator": "$trade-submit-btn" }
      │
      ▼
AutoQA resumes with new locator — test continues without restart
```

> No test restart. No human intervention. Execution continues.

# Summary

<!-- _class: compact -->

> We replaced the human translation layer between test intent and AutoQA execution.

### Before
- QA reads spreadsheet → manually assembles xlsx script → runs AutoQA → validates xlsx report
- **30–60 minutes per test case**
- High error rate: wrong locators, missing waits, duplicate steps

### After
- AI reads spreadsheet → generates + validates JSON → runs AutoQA → structured LLM analysis of xlsx report
- **Under 1 minute per test case**
- Low error rate: deterministic validator catches structural errors before AutoQA execution

### What did not change
- TestCases.xlsx — QA still writes test cases in plain language
- AdvancedActions.xlsx — macros are still reusable across both approaches
- AutoQA (java-framework) — zero modifications
- xlsx output format — AutoQA still generates the same report
