## Lessons Learned During Run
- Approval Queue (`app-queue`) loads via 1.5s simulated latency; if data is empty after first wait, prefer reload + sleep 4s before re-reading. Component re-renders after reload may invalidate prior eval references.
- Toast messages are short-lived; capture must happen within 1–2s of trigger. For verification, prefer reading store state (api.trades status) immediately after approve.

## Strategy Changes
- For toast assertions, kick off the action and immediately read `comp.toastMessage` before sleep > 1s.
